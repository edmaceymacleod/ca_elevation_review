# Spec: Schema-validation tooling + CI hardening + dev ergonomics ("tooling")

Owner work item: **tooling**. Branch in flight: `claude/codebase-capabilities-0s8u05`.
Status: **build-ready** (adversarial review resolved — see §6).

## 0. Problem statement / current state

The repo already has good bones:

- `engine/tools/validate_schemas.py` compiles every `*.schema.json` (draft-07) and
  validates discovered fixtures by filename-suffix → schema mapping. It already
  covers the committed manifest/capture/golden under `engine/fixtures/`. It is today a
  **zero-`ca_elevation_engine`-dependency** script (only `jsonschema`).
- `.github/workflows/ci.yml` runs a path-filtered `schema` job that calls
  `validate_schemas.py -v`, plus the `engine` job (ruff lint + `ruff format --check`
  + `mypy` + `pytest -m "not heavy"`).
- `.pre-commit-config.yaml` mirrors CI; `validate-schemas` runs at `pre-push`/`manual`.
- The F-01 fixture is regenerated by `engine/fixtures/seeders/f01_synthetic_office.py`
  via its `write(out_dir)` / `main()` — BUT that seeder only writes the
  **manifest + capture** into `fixtures/synthetic/`. It does **not** regenerate the
  **golden** `fixtures/golden/f01_verdict_report.json`. The golden is produced by
  running the pipeline with a fixed `generated_at="2026-06-28T00:00:00Z"` and serialized
  by the engine's own `report/json_report.py::render_json` (verified:
  `render_json(report) + "\n"` is **byte-identical** to the committed golden today,
  modulo `engine_version`).

### Verified facts that shape this spec (re-checked against the tree)

- The committed golden is serialized with `json.dumps(..., indent=2, sort_keys=True)`
  **plus a trailing `\n`** (last byte is `\n`; top-level keys are alphabetical:
  `device_results, engine_version, generated_at, project_id, schema_version, summary,
  units`). Source: `render_json`.
- The committed `synthetic/*.json` (`f01_office.manifest.json`, `f01_office.capture.json`)
  are serialized with `json.dumps(..., indent=2)` and **no trailing newline** (last byte
  is `}`) and preserve **authored key order** (NOT alphabetical).
- `run_pipeline(...).report` (a `PipelineResult` dataclass) is the report object;
  `.report.to_dict()` is the dict the integration test compares.
- `registry.SCENARIO_GOLDENS` maps `scenario_key -> golden_filename`
  (`"f01_synthetic_office" -> "f01_verdict_report.json"`). It does **not** encode the
  synthetic manifest/capture stems (those are hardcoded `f01_office.*` in the seeder),
  so a scenario→synthetic-filename link is **not** programmatically derivable.
- `CONTRIBUTING.md` line 44 says mypy is "non-blocking in CI"; `ci.yml` runs `mypy` as a
  plain blocking step. Doc-drift is real.
- The `schema` paths-filter omits `engine/tools/regen_fixtures.py`. `engine/fixtures/**`
  already covers `seeders/**`.
- The `integration` pytest marker is registered in `engine/pyproject.toml`.
- Engine version `0.1.0` matches the golden's `engine_version`.

### Concrete gaps this spec closes

1. **No fixture/golden drift guard.** A contributor can hand-edit
   `fixtures/synthetic/*.json` or let the seeder logic drift from the committed bytes
   and nothing fails. The golden has a determinism *test*
   (`test_pipeline_reproduces_golden`) but the **input fixtures** (manifest/capture) have
   no "re-run the seeder, assert no diff" guard, and there is **no single regeneration
   entry point** that also (re)writes the golden.
2. **No CLI/programmatic way to regenerate the golden.** Regeneration is tribal
   knowledge. There is no script that regenerates the *complete* fixture set (manifest,
   capture, golden) deterministically.
3. **Doc/CI drift on mypy gating.** Fix the doc to say **blocking** (CI already is).
4. **`schema` CI job doesn't see `regen_fixtures.py` changes.** Its `dorny/paths-filter`
   glob omits the new tool, so editing it won't trigger the drift validation.
5. **No bytes-level determinism contract for the seeder output.** The seeder uses
   `json.dumps(..., indent=2)` with no trailing newline for the synthetic files. The
   committed files must match the seeder's exact serialization or the new drift guard
   is unusable; we lock the serialization in one helper **per file kind** (authored-order
   for manifest/capture, engine-canonical for the golden).

### Explicit non-goals

- No changes to `revit-addin/` or `ios-app/` build steps, or their CI jobs.
- Do **not** touch/merge/alter PR #7 or any other open branch.
- No new heavy/native deps. `jsonschema` + the core pipeline (already core) is all the
  new tooling needs. Nothing behind `[heavy]` is added; no `@pytest.mark.heavy` tests
  are required (the drift guard runs in the default light suite).
- No new fixtures/scenarios (other work items). We harden the *tooling* around the
  existing F-01 scenario, written generically so adding F-02 needs no tooling change.
- Not changing report content, schema content, or engine math. In particular the golden
  bytes MUST remain byte-identical after this work (see §6, MAJOR-2).

## 2. Concrete behavior / signatures

### 2.1 `engine/fixtures/seeders/f01_synthetic_office.py` (change)

Two serializers, one per file kind, because the committed bytes are NOT uniform
(verified):

- **manifest + capture** → authored key order, `indent=2`, trailing newline.
- **golden** → the engine's own `render_json(report) + "\n"` (sorted keys, `indent=2`,
  trailing newline). This is the function that produced the committed bytes; routing the
  golden through it guarantees the seeder's golden tracks the engine serializer forever.
  (This resolves BLOCKER-1 — see §6.)

```python
from __future__ import annotations

import json
import tempfile
from pathlib import Path

# Fixed clock used to stamp the golden deterministically. MUST match the value
# tests/test_integration_golden.py uses (GENERATED_AT) so the golden the seeder
# writes is the golden the integration test reproduces. A test in
# test_fixture_drift.py asserts the two literals agree.
GENERATED_AT = "2026-06-28T00:00:00Z"


def _write_authored(path: Path, obj: dict) -> None:
    """Serializer for hand-authored fixtures (manifest, capture).

    Preserves authored key order (sort_keys=False), indent=2, UTF-8, trailing
    newline. This MUST equal the committed manifest/capture bytes after the
    one-time trailing-newline reformat (see IMPORTANT note below).
    """
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(json.dumps(obj, indent=2) + "\n", encoding="utf-8")


def _write_golden(path: Path, report) -> None:
    """Serializer for the golden: the engine's own render_json + newline.

    Byte-identical to the committed golden (sorted keys, indent=2, trailing
    newline). Do NOT route the golden through _write_authored — the committed
    golden is key-sorted, an authored-order dump would churn it.
    """
    from ca_elevation_engine.report.json_report import render_json

    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(render_json(report) + "\n", encoding="utf-8")
```

> **IMPORTANT — one-time reformat is SYNTHETIC-ONLY.** The currently committed
> `synthetic/*.json` were written with `json.dumps(indent=2)` and **no trailing
> newline** (last byte `}`). As part of this change the implementer MUST re-run the
> seeder in write mode so those two files gain a trailing `\n`. The **golden is already
> canonical** (sorted + trailing `\n`) and MUST remain byte-identical — it should show
> **no diff at all**. Verify with `git diff --stat`: exactly two files
> (`f01_office.manifest.json`, `f01_office.capture.json`) change, each by a single
> trailing-newline insertion; `f01_verdict_report.json` is unchanged. If the golden
> shows a large key-order churn, the serializer wiring is wrong (you routed it through
> `_write_authored`) — stop and fix. Commit this whitespace-only reformat together with
> the seeder change; keep it inside THIS work item's commits, do not let it bleed into
> PR #7's files.

Add `build_golden()`:

```python
def build_golden():
    """Produce the F-01 golden report object by running the engine on the
    in-memory manifest/capture, stamped with the fixed GENERATED_AT clock.

    Returns the VerdictReport (the report OBJECT, not its dict/bytes) so the
    caller serializes it via the engine's render_json. Uses the real pipeline so
    the golden can never silently diverge from engine behavior; the integration
    test independently re-derives it as a cross-check (it imports nothing from
    this seeder).

    Ingest does NOT load the referenced image/depth asset files (it only checks
    the JSON paths it is given), so no asset bytes are needed in the tempdir.
    """
    from ca_elevation_engine.pipeline import run_pipeline

    with tempfile.TemporaryDirectory() as td:
        tdp = Path(td)
        _write_authored(tdp / "m.manifest.json", build_manifest())
        _write_authored(tdp / "c.capture.json", build_capture())
        result = run_pipeline(
            tdp / "m.manifest.json",
            tdp / "c.capture.json",
            generated_at=GENERATED_AT,
        )
    return result.report
```

Extend `write()` to also emit the golden, and add a `golden` key to the returned dict.
`out_dir` keeps meaning the *synthetic* dir for backward compat; the golden dir is
resolved relative to it as `out_dir.parent / "golden"`. **This `out_dir.parent/"golden"`
contract is load-bearing and must be honored by every future seeder** so that
`regen_fixtures.py` (which calls `write()` into a tempdir mirroring
`<td>/synthetic` + `<td>/golden`) can map outputs to committed paths by relative path
without re-deriving them (see §2.2, MAJOR-1):

```python
def write(out_dir: Path) -> dict[str, Path]:
    out_dir.mkdir(parents=True, exist_ok=True)
    manifest_path = out_dir / "f01_office.manifest.json"
    capture_path = out_dir / "f01_office.capture.json"
    golden_path = out_dir.parent / "golden" / "f01_verdict_report.json"
    _write_authored(manifest_path, build_manifest())
    _write_authored(capture_path, build_capture())
    _write_golden(golden_path, build_golden())
    return {"manifest": manifest_path, "capture": capture_path, "golden": golden_path}
```

`main()` unchanged in shape (prints each written path; now includes `golden`).

Rationale for `build_golden()` calling the real pipeline: the existing integration test
already asserts pipeline-output == committed golden, so the golden is by definition "the
engine's output on this fixture". Deriving it the same way means **one** regeneration
command produces a self-consistent set, while the integration test remains an
independent re-derivation.

### 2.2 `engine/tools/regen_fixtures.py` (add)

Generic over scenarios via the registry, so future fixtures need no edit here.

```python
#!/usr/bin/env python3
"""Regenerate (or drift-check) every committed fixture from its seeder.

Walks ca_elevation_engine.registry.SCENARIOS, loads each seeder module from
engine/fixtures/seeders/<scenario>.py BY FILE PATH (fixtures/ is not a package on
sys.path), and calls its write() to (re)produce the committed manifest, capture,
and golden. Deterministic: no clocks, no randomness (the golden clock is the
seeder's fixed GENERATED_AT).

The serializer inside the seeder is the single authority for all three files'
bytes; this tool only diffs committed bytes against what the seeder writes.

Modes:
    (default)        rewrite the committed fixtures in place.
    --check          DO NOT write; regenerate into a tempdir and byte-diff against
                     the committed files. Exit 1 (with a unified diff to stderr) on
                     any drift. This is the CI / pre-push guard.

Options:
    --fixtures-root DIR   committed-fixtures root to compare/write against
                          (default: REPO_ROOT/engine/fixtures). Lets tests target a
                          tmp copy.
    -v / --verbose

Exit codes:
    0  all fixtures regenerated (or, with --check, all match)
    1  --check found drift, a seeder/registry inconsistency, or the engine import
       failed (regen runs the pipeline; install with `pip install -e engine`)
"""
```

Module constants (required so tests can redirect — resolves MINOR-1):

```python
REPO_ROOT = Path(__file__).resolve().parents[2]
SEEDERS_DIR = REPO_ROOT / "engine" / "fixtures" / "seeders"
FIXTURES_ROOT = REPO_ROOT / "engine" / "fixtures"   # default committed root
```

Required functions:

- `def _seeder_module(scenario: str) -> ModuleType`
  Load `SEEDERS_DIR / f"{scenario}.py"` via `importlib.util.spec_from_file_location`.
  Fail with a clear message (`f"seeder not found for scenario {scenario!r}: {path}"`,
  exit 1) if the file is missing — mirrors `test_registry`'s file-exists assertion.
  **No package import / no `__init__.py`** is used (resolves MINOR-3).

- `def regenerate(scenario: str, *, check: bool, fixtures_root: Path = FIXTURES_ROOT,
  verbose: bool = False) -> list[str]`
  Call the seeder's `write()` into a tempdir laid out as `<td>/synthetic` (passed as
  `out_dir`), which — by the §2.1 `out_dir.parent/"golden"` contract — also produces
  `<td>/golden`. Then for each produced file, map it to its committed counterpart under
  `fixtures_root` by its path **relative to the tempdir root** (e.g.
  `synthetic/f01_office.manifest.json`, `golden/f01_verdict_report.json`). Compare
  `produced.read_bytes()` vs `committed.read_bytes()` **byte-exact**.
  - In **check** mode: return a list of human-readable drift messages (one per
    drifted-or-missing committed file), empty when clean. Do not write.
  - In **write** mode: copy produced bytes onto the committed paths; return `[]`.
  Reusing `write()` for path logic guarantees regen and the seeder can never disagree
  on filenames or serialization (resolves MAJOR-1). Note: this does tempdir I/O purely
  to read bytes back — acceptable for a maintainer tool.

- `def main(argv: list[str] | None = None) -> int`
  argparse with `--check` (store_true), `--fixtures-root` (type=Path, default
  `FIXTURES_ROOT`), `-v/--verbose`. Iterate `registry.SCENARIOS`. If importing the
  registry / running a seeder raises `ImportError`, print
  `ERROR: install the engine first: pip install -e engine` to stderr and return 1.
  Aggregate drift across scenarios. On `--check` with drift: for each drifted file print
  a `difflib.unified_diff` (text-decoded, with `committed`/`regenerated` labels) to
  stderr, then `return 1`. On success print `Fixtures up to date.` (check) or
  `Regenerated N file(s).` (write); `return 0`.

The committed-bytes comparison MUST be byte-exact (`read_bytes()`), since the whole
point is catching whitespace/ordering drift.

### 2.3 `engine/tools/validate_schemas.py` (change)

Two small, backward-compatible hardenings. Note up front (and add one sentence to the
script docstring): the script's schema/fixture-validation **core stays engine-free**
(only `jsonschema`); the new orphan check adds a *guarded, best-effort* `registry`
import that is used only when the engine is importable (resolves MINOR-5).

1. **`--strict-unknown` flag — opt-in (default False), CI passes it.** Add
   `parser.add_argument("--strict-unknown", action="store_true")` and thread it as
   `validate_fixtures(fixtures_dir, schemas, verbose, *, strict_unknown=False)`. When
   `strict_unknown` is True, the existing `skipped` list (unrecognized `*.json`) becomes
   a **hard error** appended to `errors` instead of only a NOTE — so a typo'd
   `f02_office.manfest.json` cannot silently escape validation. When False, behavior is
   exactly as today (NOTE only). There is no "default True in CI" mode — the flag is
   plain opt-in and CI passes it explicitly (resolves MAJOR-3). **It is a no-op on
   today's tree** (zero unrecognized `*.json` under `fixtures/`); its only value is
   guarding future misnamed payloads.

2. **Orphan-golden check — "every registered golden exists", engine-optional.** After
   fixture validation, attempt `from ca_elevation_engine import registry` inside
   `try/except ImportError`. If it imports, assert that **every** golden filename in
   `registry.SCENARIO_GOLDENS.values()` exists somewhere under `fixtures_dir` (i.e. a
   registered golden was not deleted); append an error naming any missing golden. If the
   import fails, print a NOTE that the registry cross-check was skipped and continue
   (degrade gracefully for standalone runs).
   **Scope is deliberately "golden exists" only** — NOT "a matching manifest/capture
   exists" — because the registry maps `scenario -> golden_filename` and does **not**
   encode the synthetic `f01_office.*` stems, so a generic input-match is not derivable
   from the data model (resolves MINOR-2). The real win this closes is catching a deleted
   /renamed golden, which the registry *can* express.

No change to schema compilation logic or the suffix map.

## 3. Test plan (all headless, Linux, no Revit/heavy)

New file `engine/tests/test_fixture_drift.py`, marked `pytest.mark.integration` (runs in
default `-m "not heavy"` CI):

1. **`test_committed_fixtures_match_seeders`** — for every `registry.SCENARIOS`, assert
   `regen_fixtures.regenerate(scenario, check=True) == []`. In-process twin of the CI
   `--check` step; catches drift even when the path-filter wouldn't have triggered the
   `schema` job.
2. **`test_regen_check_main_exit_zero`** — `assert regen_fixtures.main(["--check"]) == 0`
   on the clean tree.
3. **`test_regen_detects_injected_drift`** — copy the real `fixtures/` tree into
   `tmp_path` (`shutil.copytree`), mutate one byte of
   `synthetic/f01_office.capture.json`, then call
   `regen_fixtures.main(["--check", "--fixtures-root", str(tmp_copy)])` (or
   `regenerate(scenario, check=True, fixtures_root=tmp_copy)`) and assert it returns 1 /
   a non-empty drift list whose message names the file. Confirms the guard fails closed.
4. **`test_seeder_golden_bytes_match_committed`** — call the seeder's `build_golden()`,
   serialize via `render_json(report) + "\n"`, and assert it equals the committed golden
   **bytes** (modulo nothing — fixed clock + real version). Cross-checks the seeder's
   golden derivation equals the committed bytes exactly (this is the byte-level
   complement to the integration test's dict-level comparison).
5. **`test_synthetic_serialization_is_canonical`** — for `synthetic/*.json`: assert each
   ends with exactly one trailing `\n` and round-trips through `_write_authored` to
   identical bytes. For the golden: assert it ends with exactly one `\n` and equals
   `render_json(report)+"\n"`. Locks the per-kind serialization contract.
6. **`test_generated_at_constants_agree`** — assert
   `seeder.GENERATED_AT == test_integration_golden.GENERATED_AT` (import the test
   module's constant) so the two literals can't drift.

New file `engine/tests/test_validate_schemas_tool.py` (marked `unit`):

7. **`test_strict_unknown_fails_on_misnamed_payload`** — build a tmp fixtures dir with a

  engine test suite, both Linux-hosted.
- **No heavy backends:** `regen_fixtures.py`'s golden derivation uses only the core
  pipeline + `render_json` (no open3d/pye57/opencv, no reportlab — the golden is the
  JSON report, not a PDF). The drift test is plain `integration`, not `heavy`, so it runs
  in default CI. No `@pytest.mark.heavy` is introduced.
- **jsonschema/engine absent (ad-hoc standalone `validate_schemas.py`):** the registry
  orphan cross-check is wrapped in `try/except ImportError` and degrades to a NOTE; the
  schema/fixture validation core stays engine-free. `regen_fixtures.py` hard-requires the
  engine (it runs the pipeline) and prints
  `ERROR: install the engine first: pip install -e engine` and exits 1 if absent —
  acceptable because regen is a maintainer task.
- **Determinism:** the only clock is the seeder's fixed `GENERATED_AT`; `engine_version`
  is stamped from the installed package version. If the package version is bumped, the
  golden's `engine_version` changes and `regen_fixtures.py` (write mode) updates it; the
  `--check` guard then correctly flags a stale committed golden until regenerated — this
  is desired (forces a deliberate, reviewed golden bump on version changes), and is
  documented in CONTRIBUTING. The integration test strips `engine_version` before
  comparing, so it won't double-fail on a version bump.

## 5. Risks / call-outs

- **Trailing-newline reformat is SYNTHETIC-ONLY and required once.** If skipped, the
  drift guard fails on a clean tree. Mitigate: regen in write mode first; verify
  `git diff --stat` shows exactly the two `synthetic/*.json` changed by one newline each
  and the golden unchanged. Keep it in THIS work item's commits; do not let it bleed into
  PR #7.
- **Golden must stay byte-identical.** Because the golden is already canonical, any diff
  to `f01_verdict_report.json` from this work indicates a serializer-wiring bug (golden
  routed through `_write_authored`). The §2.1 verification step catches it.
- **`engine_version` coupling:** any version bump makes `--check` fail until regen.
  Documented as intended; the integration test strips `engine_version`, so no double-fail.
- **Seeder `out_dir.parent/"golden"` contract is load-bearing** for `regen_fixtures.py`'s
  tempdir-mirror mapping. Every future seeder must honor it; stated as an explicit
  invariant in §2.1/§2.2.
- **Do not touch other branches/PRs:** all edits are confined to the files in §1, all on
  the engine/tooling surface owned by this work item.

## 6. Adversarial review resolutions

Every BLOCKER and MAJOR from `spec-tooling.review.md` is resolved below; cheap MINORs are
resolved too. Where the reviewer offered options, the chosen option and rationale are
stated.

- **BLOCKER-1 — `sort_keys=False` would not reproduce the committed golden.**
  RESOLVED via the reviewer's preferred option (a). The single author-order serializer is
  split into two: `_write_authored` (manifest/capture, `sort_keys=False`) and
  `_write_golden`, which serializes via the engine's own
  `render_json(report) + "\n"` (sorted keys) — the exact function that produced the
  committed golden (verified byte-identical). The golden is never routed through the
  authored-order path. See §2.1. Rationale for (a) over a `sort_keys` parameter: it makes
  the seeder's golden bytes track `render_json` automatically if the engine serializer
  ever changes indent/sort.

- **MAJOR-1 — `regenerate()` reuse-via-`write()` underspecified / coupled to the
  serializer bug.** RESOLVED. With BLOCKER-1 fixed at the serializer, reusing `write()`
  now reproduces correct bytes. §2.2 now explicitly states: regen diffs committed bytes
  against the seeder's serializer output (the serializer is the authority for all three
  files), and documents the `out_dir.parent/"golden"` tempdir-mirror contract as
  load-bearing (also called out in §2.1 and §5).

- **MAJOR-2 — "one-time trailing-newline reformat" is wrong for the golden.** RESOLVED.
  §0, §2.1, and §5 now state the reformat is **synthetic-only**; the golden already ends
  in `\n` and MUST remain byte-identical. The `git diff --stat` check is restated to
  expect exactly the two synthetic files changed and the golden unchanged, with an
  explicit "stop and fix" if the golden churns. (Verified against the tree: golden last
  byte is `\n`, synthetic last byte is `}`.)

- **MAJOR-3 — `--strict-unknown` "default True in CI" is self-contradictory.** RESOLVED.
  The "default True in CI" clause is deleted. §2.3 defines the flag as plain opt-in
  (`action="store_true"`, default False); CI passes it explicitly. Also states it is a
  no-op on today's tree (zero unrecognized `*.json`), guarding future typos only.

- **MAJOR-4 — pre-commit `fixture-drift` keyed off `src/ca_elevation_engine/` is broad
  and slow.** RESOLVED. §2.5 narrows the pre-commit trigger to `^engine/(fixtures/|
  tools/regen_fixtures\.py)`. The engine-change-moves-golden case is already covered by
  `test_integration_golden.py::test_pipeline_reproduces_golden` (test suite) and by CI's
  `fixture-drift` step; the local pre-push hook does not need the broad `src/**` glob.

- **MINOR-1 — Test #3 needs an injection point the design didn't specify.** RESOLVED.
  §2.2 adds module constant `FIXTURES_ROOT` and a `--fixtures-root` argparse option plus a
  `fixtures_root` param on `regenerate()`. Test #3 (§3) uses `shutil.copytree` into
  `tmp_path` and points regen at the copy.

- **MINOR-2 — orphan-golden cross-check can't bind manifest/capture to a scenario.**
  RESOLVED via the reviewer's lower-risk option (a). §2.3 scopes the check to "every
  registered golden filename exists" only (catches a deleted/renamed golden); it does NOT
  assert a matching manifest/capture, because the registry does not encode the synthetic
  stems.

- **MINOR-3 — `__init__.py` in `fixtures/seeders/` is unnecessary.** RESOLVED. The
  `__init__.py` add is dropped from §1; §2.2 commits to file-path import via
  `spec_from_file_location` only.

- **MINOR-4 — redundant `seeders/**` paths-filter line.** RESOLVED. §2.4 adds only the
  load-bearing `engine/tools/regen_fixtures.py` line; the redundant `seeders/**` line is
  dropped.

- **MINOR-5 — `validate_schemas` lazy registry import is a coupling change.** RESOLVED.
  §2.3 states (and the script docstring will note) that the validation core stays
  engine-free; the orphan check adds a guarded, best-effort, engine-optional `registry`
  import used only when the engine is importable, degrading to a NOTE otherwise.

No findings are disputed; all are adopted. Where the review presented A/B options
(BLOCKER-1, MINOR-2), the chosen option and rationale are recorded above.
